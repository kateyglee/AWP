Design System · v3

AWP Design System

The token system that powers AWP — from primitive color scales through semantic aliases to chart palettes. Source of truth for designers and engineers building the publisher-facing AI agent for programmatic ad ops.

The system in one paragraph
AWP uses a two-tier token architecture. Primitives are the raw color/spacing/radius scales — never referenced in components. Aliases are semantic role-based names like --text-dk, --surface-card, --button-bg-primary. Components reference only aliases. Light and dark modes share alias names but point at different primitives, so the same component code renders correctly in both modes.

01
Architecture & principles
Two tiers of tokens. Components only ever reference the alias layer — that's what makes light/dark mode and future brand portability possible without re-touching components.
01
Primitives
Raw scales
Numbered color ramps (--color-brand-sand-100…950), spacing increments, radius steps, breakpoints. These are the design system's vocabulary, but components never reference them directly. Reaching for a primitive in component CSS breaks dark mode and ties the component to a specific palette.
02
Aliases
Semantic roles
Role-based names: --text-dk, --surface-card, --tag-bg-pending, --button-bg-primary. Each alias maps to a primitive in light mode and (usually) a different primitive in dark mode. Components reference only aliases.
03
Modes
Theme switch
Light mode is the default at :root. Dark mode overrides aliases under [data-theme="dark"]. Theme switching swaps alias values; component CSS is identical across modes.
Principle
Aliases are the contract
Components reach for --surface-card and --text-dk, never --color-brand-sand-50 or --color-brand-slate-700. The alias layer is what makes light/dark mode work, what keeps the system theme-able, and what decouples engineering from the primitive palette. If you reach for a primitive in component code, you've broken the contract.
Principle
Color is reserved for state and data
Surfaces and text are warm/cool neutrals. Saturated color appears only on status pills, impact tags, diff additions, errors, and chart data. The interface looks calm because color carries meaning, never decoration. When the agent surfaces a chart full of color, it pulls focus by contrast with the quiet shell around it.
Principle
Two type families, two roles
Satoshi for prose and UI — titles, body copy, button labels, AI conversational text. JetBrains Mono for data and system values — numbers, timestamps, IDs, paths, hex codes, status pills. The split is by content type, not by author. When a publisher sees mono inline in a sentence, they know "this is a computed value."
02

Amber, Green, Red, Blue, Violet carry semantic meaning. Teal, Steel-Blue, Lavender, Dusty-Teal, Blush support data viz patterns (categorical anchors, monochromatic ramps, diverging palettes). Family rules
  • Family is automatic. Tokens in the prose/UI group always render in Satoshi; tokens in the data/system group always render in JetBrains Mono. Components reference the role; the system enforces the family.
  • The split is meaningful. When a publisher sees JetBrains Mono, they know "this is computed by the system or pulled from data." When they see Satoshi, they know "this is presented to me by the product."
  • Caps treatment is part of the role. Pill, Eyebrow, and Label SM are always uppercase — the typography token carries the case transformation.

Implementation notes
  • Reference aliases only. Component CSS should never reference primitives like --color-brand-sand-700. Reaching for primitives breaks dark mode and ties components to a specific palette.
  • Modal scrim = surface-overlay + backdrop-blur. The two tokens function as a single pattern. Modal backdrops always use both: background: var(--surface-overlay) for the dim, backdrop-filter: var(--backdrop-blur) for the glassmorphism. They're separate tokens only because they apply to different CSS properties — there's no scenario in AWP where you use one without the other.
  • Tertiary buttons are CSS-only conveniences. --button-bg-tertiary resolves to transparent — Figma can't hold "transparent" as a paint variable, so this token lives in CSS only. The hover, text, and disabled states all have proper Figma variables.
  • Icon tokens use Brand-Slate in both modes. Icons match text legibility, regardless of whether the surrounding surface is warm Sand or cool Slate.
  • Shadow tokens are CSS-only. Figma's Effect Styles don't auto-swap with Variable modes. Designers maintain parallel Effect Styles named shadow/lt/sm, shadow/lt/md, etc., and swap manually when toggling modes.

What changes between modes
  • Surfaces flip neutral families. Light mode surfaces use Brand-Sand (warm); dark mode surfaces use Brand-Slate (cool). The structural hierarchy (canvas → base → raised → elevated → card) is preserved in both.
  • Text stays on Brand-Slate in both modes. Slate gives more consistent legibility against both warm and cool backgrounds than swapping text families would.
  • Semantic state colors deepen in dark mode. Pending/success/error/info backgrounds shift from -100 tints in light mode to -900 tones in dark mode. Text on those surfaces inverts to -300.
  • Buttons invert. Primary buttons go dark-on-light in light mode; light-on-dark in dark mode.
  • Components don't change. The proposal card above uses identical CSS — only the alias values resolve to different primitives.

07
Implementation
How to apply the system in code. The pattern is the same whether you're writing component CSS, Figma variables, or Tailwind utilities — reference the alias name; let the system resolve the primitive.
Component CSS — the only pattern
/* GOOD — references aliases, works in both modes */ .proposal-card { background: var(--surface-card); color: var(--text-dk); border: 1px solid var(--border-stroke); border-radius: var(--radius-xl); padding: var(--spacing-4) var(--spacing-5); } /* BAD — references a primitive directly. Breaks dark mode. */ .proposal-card { background: var(--color-white); /* don't do this */ color: var(--color-brand-slate-900); /* don't do this */ }
Activating dark mode
/* HTML: add data-theme on the root element */ <html data-theme="dark">...</html> /* JS: toggle the attribute */ document.documentElement.dataset.theme = isDark ? 'dark' : 'light';
Source-of-truth contract
  • Figma variables and CSS variables stay 1:1. Every primitive and alias in this doc has a matching Figma variable with the same name. When tokens change, both update together.
  • Four exceptions live in CSS only. --button-bg-tertiary (transparent, can't be a Figma paint variable), --surface-overlay (rgba with alpha — Figma variables don't carry their own opacity), --shadow-* (composites — Figma uses parallel Effect Styles that swap manually), and --backdrop-blur (filter, not paint) — all called out in their respective sections.
  • Designers reach for the alias name first. If no alias exists for the role you need, ask whether a new alias should be added rather than reaching for a primitive.
  • Engineers ship with aliases too. Tailwind utilities like bg-surface-card or text-dk map directly through the same custom property layer.
A complete chart palette system
AWP's data viz system is structured around four palette types — categorical (12 colors in strict sequence), monochromatic (single-hue intensity scales), diverging (two opposing scales meeting at a neutral center), and status (semantic state, separate from data). Inspired by IBM Carbon's palette framework and adapted to AWP's neutral system. Categorical and diverging palettes have dedicated alias tokens. Monochromatic and recommended combinations use primitive references directly — they're patterns, not separate tokens.
Sections 01 · Architecture 02 · Categorical 03 · Monochromatic 04 · Diverging 05 · Combinations 06 · Status
01
Architecture
Four palette types, each with a job. Categorical separates unrelated series. Monochromatic shows intensity within a single dimension. Diverging shows magnitude in two directions from a neutral center. Status carries semantic state and stays out of categorical use. Picking the right palette is the most important decision in any chart.
Palette · 01
Categorical
12 colors applied in strict sequence. Use when each series is unrelated and equally weighted (bidder names, ad units, geos). Stops at 12 — past that the palette breaks down and you need a different chart strategy.
Palette · 02
Monochromatic
Single-hue ramps for charts where intensity matters more than category — heatmaps, density plots, ordinal data. Four hues to choose from: Slate (default), Blue, Teal, Violet.
Palette · 03
Diverging
Two opposing scales meeting at a neutral center. Two options: blush-steel (carries temperature) and lavender-dusty (no temperature). 7 steps each.
Palette · 04
Status
Reserved for chart elements that carry semantic state — annotation lines, alert markers, KPI sparkline directionality. Never used as a series color in a categorical chart.
Ground rules
  • Monochromatic is not categorical with opacity. Mono palettes use primitive ramp steps directly (e.g., blue-100 through blue-700), not transparency. Opacity introduces visual noise from underlying surface tone.
  • Categorical sequence is non-negotiable. Always start at slot 1. Two-series charts use slots 1 + 2. Don't skip slots, don't reorder, don't reuse out of sequence.
  • There is no slot 13. The categorical palette stops at 12. When data has more than 12 distinct categories, the chart needs a different strategy — not more colors. Reusing or generating new categorical colors past slot 12 causes false meaning collisions.
02
Categorical sequence
12 colors, applied in strict order. The palette is intentionally desaturated — chalky and dusty rather than vivid — so that charts feel calm against AWP's warm-Sand light surface and cool-Slate dark surface. Slots 1, 6, 7, 8 reference dedicated primitive ramps (steel-blue, lavender, dusty-teal, blush) because they're also used by monochromatic and diverging palettes. Slots 2, 3, 4, 5, 9, 10 are direct hex values — they're only used as categorical anchors. Slots 11, 12 reference existing brand-slate primitives. Same values render in both light and dark mode — the luminance works on both canvases. Single-series charts use --viz-cat-1; two-series adds slot 2; etc. The sequence is non-negotiable.

How to apply categorical
  • Always start at slot 1. Single-series charts use --viz-cat-1. Two-series adds slot 2. Etc. The sequence guarantees adequate neighbor contrast — skipping or reordering breaks that guarantee.
  • The order is the contrast structure. Slot 1 (steel-blue) is intentionally slightly more saturated than the rest — it gives the primary series visual weight. Slots 2–10 sit at uniform desaturation; slots 11–12 are neutrals.
Which slots reference primitives, and why
  • Slots 1, 6, 7, 8 reference dedicated primitive ramps (steel-blue, lavender, dusty-teal, blush). They earn primitives because something downstream — monochromatic palettes, diverging palettes — needs multiple steps from the same hue ramp, not just the single anchor value.
  • Slots 2, 3, 4, 5, 9, 10 stay as direct hex values. They're used only as their categorical anchor — nothing else in the data viz system pulls from those hue ramps. Promoting them to primitives would create 60+ unused tokens.
  • Slots 11, 12 reference existing brand-slate primitives. Slate is structural and is correctly reused across the system; the neutrals stay tied to the system rather than duplicated.
  • Mnemonic names (steel-blue, sage, terracotta, etc.) live in primitive scale names and CSS comments only. They are not exposed as cat alias names. The categorical contract is positional — --viz-cat-1 through --viz-cat-12 — because position is the contract, not the hue.
  • Same values in light and dark mode. The desaturated palette's luminance (~65–75%) reads correctly against both warm-Sand light surface and cool-Slate dark surface, so no mode override is needed.

How monochromatic works
  • The pattern is the contract, not a token name. A mono palette is "steps 100 through 700 of a sanctioned primitive ramp." Encoding that as 28 alias tokens (4 hues × 7 steps) would duplicate the primitives with zero semantic benefit.
  • Pull from primitives directly in chart code. Reference --color-steel-blue-100 through --color-steel-blue-700 (or whichever sanctioned ramp). This is the one place in the system where component code can reach for primitives.
  • Use Slate by default. When the chart subject doesn't suggest a hue, use Slate. It carries no associations and works in both light and dark modes without re-tuning.
  • Step direction depends on encoding. For "more is more important" (heatmap intensity), darkest step represents the highest value. For "less is less data" (density plots), lightest step is sparse data.

When to use which
  • Blush-Steel carries temperature. Use it when the data has a clear good/bad or above/below interpretation: bid floor performance vs benchmark, eCPM lift vs prior period, fill rate change. Blush ends read as "underperforming," steel-blue ends as "outperforming."
  • Lavender-Dusty is neutral. Use it when the data has two valid sides without good/bad weighting: device split (mobile vs desktop), bidder share (Bidder A vs Bidder B), audience segmentation. Neither end carries an emotional charge.
  • Neutral center is the fixed pivot. Both palettes use a slate-tone center step (slate-200 in light mode, slate-700 in dark mode). The center is where "no movement" reads — symmetric distance to either end means symmetric magnitude.
  • Diverging is not a substitute for two-series categorical. If the two sides are unrelated categories (not magnitudes from a center), use categorical slots 1 and 2 instead.

Categorical vs combinations — which to use
  • Categorical sequence is the default. Reach for --viz-cat-1, --viz-cat-2, etc. when you want the standard neighbor contrast across any series count up to 12.
  • Combinations are tested groupings. Use them when you want a known-balanced pairing — particularly when the data is small (2–4 series) and the chart is supporting context, not the page's focus.

  • Don't mix combinations and sequence. Pick one approach for the chart — don't combine combo slots with non-combo cat slots in the same chart.

Status colors
Reserved for chart elements that carry semantic state — annotation lines, alert markers, threshold indicators, KPI sparkline directionality (up/down trend coloring). These exist as separate alias tokens distinct from --tag-bg-*; the status alias values are the specific shades that carry state meaning across the entire system. Never used as a categorical series color.

How to apply status in charts
  • Annotations and markers. A threshold line crossing a chart, an alert dot on a sparkline, a benchmark band — these use status tokens. Success for "above target," warning for "approaching limit," error for "exceeded threshold," info for neutral reference.
  • Sparkline directionality. KPI cards that color the sparkline by trend (up = success, down = error) reference the status tokens. The sparkline's up-trend isn't a "green series" — it's a status indicator.
  • Reserved values, not reserved hues. The status alias values (green-700, amber-700, red-500, blue-600) are reserved across the system. Other shades from those hue families — like green-600 in categorical slot 7, or yellow-500 in slot 6 — can appear in charts because they're different shades and Yellow is a different hue from Amber.
  • Avoid hue collision in alerting contexts. When a chart's subject is performance, errors, or status itself, prefer categorical slots that don't share a hue family with status colors. This is judgment, not a hard rule — but a "bidder performance" chart that uses green-600 in slot 7 should pause and consider whether it might confuse readers.
  • Status values stay constant across modes. Unlike most aliases, status colors don't shift between light and dark mode. Their meaning is absolute — green is success in any context.
